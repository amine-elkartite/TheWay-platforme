# Security Policy

## Overview

TheWay takes security seriously. This document explains how security issues should be reported and which versions of the project are currently supported.

Please do not publicly disclose security vulnerabilities before they have been reviewed and addressed.

---

## Supported Versions

TheWay is currently under active development.

| Version | Supported |
|---|---|
| Current development version | Yes |
| Older versions | No |

Security fixes and improvements are applied to the latest active version of the project.

---

## Reporting a Vulnerability

If you discover a security vulnerability, please report it responsibly to the project maintainer.

When reporting an issue, please include:

- A clear description of the vulnerability
- Steps to reproduce the issue
- The affected page, feature, or module
- The possible security impact
- Screenshots, logs, or proof of concept if available
- Suggested fix or recommendation, if possible

Please avoid sharing exploit details publicly before the issue is reviewed.

---

## What Not to Include Publicly

Do not publish or commit:

- Passwords
- API keys
- Access tokens
- Database credentials
- Private environment files
- Personal user data
- Exploit code that could harm users or systems

---

## Responsible Disclosure

The project maintainer will review reported vulnerabilities and work on a fix when applicable.

The goal is to protect users, improve the project, and handle security issues responsibly.

---

## Security Scope

Security reports may include issues related to:

- Authentication
- Authorization
- Admin access
- User accounts
- Uploaded files
- API routes
- Input validation
- Data exposure
- Configuration mistakes

---

## Security Best Practices

The project follows general security practices such as:

- Protecting private routes
- Validating user input
- Using role-based access control
- Keeping sensitive data out of the public repository
- Reviewing dependencies
- Avoiding exposure of internal errors
- Securing uploaded files
